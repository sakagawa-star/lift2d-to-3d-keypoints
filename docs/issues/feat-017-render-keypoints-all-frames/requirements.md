# feat-017 要求仕様書: render_keypoints.py 全フレーム対応（連番PNG + MP4）

## 1.1 プロジェクト概要

- **何を作るのか**: `phase4/render_keypoints.py` を、C3D の先頭フレーム1枚のみ描画する
  現状から、**全フレーム（または指定範囲）**を描画して**連番PNG**と**MP4**で出力できるように拡張する。
- **なぜ作るのか**: feat-015/016 は静止画1枚での目視検証用だった。動作中の人体キーポイントが
  3DGS空間に正しく重なるかを**時系列（動画）**で検証したい。feat-016 で「次案件に分割」と
  明記された機能を実装する。
- **誰が使うのか**: キャリブレーション結果（R, t, K）と人体キーポイントの整合を検証する開発者。
- **どこで使うのか**: phase4 の uv 環境（CUDA、gsplat）。`TORCH_CUDA_ARCH_LIST="9.0+PTX"` 必須。

## 1.2 用語定義

- **C3Dフレーム番号**: C3D に記録された各時刻フレームの番号。`py-c3d` の `read_frames()` が返す
  `frame_no`（1始まり）を指す。本ドキュメント・設計書・コードで同じ意味で使う。
- **背景レンダリング**: 3DGS（PLY）を gsplat でピンホール投影した RGB 画像・深度マップ・αマップ。
  カメラ固定のため全フレームで不変。
- **オーバーレイ**: 背景レンダリングのコピーにキーポイント点＋ボーンを描画した画像。
- **C3D rate**: C3D ヘッダに記録された点群サンプリングレート（`py-c3d` の `reader.point_rate`）[Hz]。

## 1.3 機能要求一覧

### FR-001 全フレーム読み込み
- **概要**: C3D の全フレームを読み込み、各フレームの Halpe26 座標と有効フラグを得る。
- **入力**: C3Dファイルパス（既存の必須引数 `c3d_path`）。
- **出力**: フレームごとの `(frame_no, kpts(26,3), valid(26,))` のリスト相当。
- **受け入れ基準**: N フレームある C3D を渡すと N フレーム分のキーポイントが取得され、
  欠損サンプル（residual < 0 または NaN）が各フレームで個別に valid=False になる。

### FR-002 フレーム範囲指定
- **概要**: `--start-frame`/`--end-frame` で描画対象の C3D フレーム番号範囲を指定する（両端含む）。
- **入力**: `--start-frame`（int, 省略時は最小フレーム）、`--end-frame`（int, 省略時は最大フレーム）。
- **出力**: 範囲内フレームのみ描画。
- **受け入れ基準**:
  - 両方省略時は全フレームを描画する。
  - `--start-frame 10 --end-frame 20` で frame_no 10〜20 の11フレームのみ描画する。
  - 範囲に該当フレームが0件のときは標準エラーにメッセージを出し終了コード1で終了する。

### FR-003 背景レンダリングの1回計算・共有
- **概要**: カメラ固定のため、背景3DGS（RGB・深度・α）を全フレームで1回だけ計算し共有する。
- **入力**: PLY、TOML、`--camera`、`--near-plane`、`--background`。
- **出力**: 全フレームで同一の背景画像・深度マップ・αマップ。
- **受け入れ基準**: gsplat の `rasterization` 呼び出しがフレーム数によらず1回であること
  （オクルージョンON時。OFF時は深度・α不要だが RGB は1回計算）。

### FR-004 各フレームのオーバーレイ描画
- **概要**: 各フレームのキーポイントを投影・可視判定し、共有背景に点＋ボーンを描画する。
- **入力**: FR-001 の各フレームキーポイント、FR-003 の背景・深度・α。
- **出力**: フレームごとのオーバーレイ画像（H,W,3 BGR uint8）。
- **受け入れ基準**: feat-016 と同じ投影・オクルージョン判定（`--no-occlusion`/`--occlusion-margin`
  も従来通り機能）が各フレームに適用される。先頭フレームの出力が feat-016 の単一PNGと一致する。

### FR-005 連番PNG出力
- **概要**: 各フレームのオーバーレイを連番PNGとして出力ディレクトリに保存する。
- **入力**: `--output-dir`（省略時 `./data/keypoints_<カメラ名>/`）。
- **出力**: `frame_<C3Dフレーム番号:06d>.png`（例 `frame_000001.png`）。
- **受け入れ基準**: 範囲内フレーム数と同数のPNGが出力され、ファイル名がC3Dフレーム番号で連番になる。
  出力ディレクトリが存在しなければ作成する。

### FR-006 MP4出力
- **概要**: `--mp4` 指定時、連番PNGに加えてMP4動画も出力する。
- **入力**: `--mp4`（フラグ）、`--mp4-fps`（**float**, 省略時はC3D rate を採用）。
- **出力**: `<output-dir>/output.mp4`。
- **受け入れ基準**:
  - `--mp4` 指定時、連番PNGとMP4の両方が出力される。
  - `--mp4-fps` 省略時、MP4のフレームレートがC3D rate と一致する。**C3D rate が 29.97 や
    59.94 のような非整数の場合も丸めず小数のまま保持**してMP4に反映する（動画の実時間が
    ずれないようにする）。C3D rate が取得不能（None・0・非数値）なら 30 を既定とし警告を出す。
  - `--mp4-fps 59.94` のような小数指定時はそれが優先される。
  - `--mp4` 指定時に ffmpeg が見つからない場合は、フレームループ開始**前**に標準エラーへ
    メッセージを出し終了コード1で終了する（この時点では連番PNGも未出力。MP4を要求したのに
    生成できないため処理自体を中止する。`--mp4` を付けなければPNGのみ正常出力される）。
  - ffmpeg がレンダリング途中で異常終了した場合（パイプ切断等）も traceback で落ちず、
    ffmpegプロセスをクリーンアップした上で標準エラーに ffmpeg の stderr を出して終了コード1で終了する。

## 1.4 非機能要求

- **パフォーマンス**: 背景レンダリングを全フレームで共有することで、フレーム数 N に対し
  gsplat 呼び出しは1回に抑える。1フレームあたりの追加コストは投影＋描画＋PNG書き込みのみ。
- **対応環境**: phase4 の uv 環境（Python 3.10、CUDA、gsplat 1.5.3）。GPU 必須（背景レンダリング）。
- **信頼性**: 連番PNGは1フレームずつ逐次保存し、途中失敗時もそれまでのPNGは残る。
- **後方互換**: feat-016 の単一PNG出力（`--output <file.png>`）は廃止する（後述 1.5）。

## 1.5 制約条件

- 使用必須: NumPy, OpenCV, tomli, py-c3d, torch, gsplat（既存）。MP4化に ffmpeg（render.py と同様）。
- 新規ライブラリ追加なし。
- **CLI変更（破壊的）**: 単一PNG用の `--output <file>` を廃止し、連番PNG用の `--output-dir <dir>`
  に置き換える。1フレームだけ確認したい場合は `--start-frame N --end-frame N` で代替する。
  - 理由: 全フレームが基本動作になるため出力はディレクトリが自然。単一PNGとディレクトリの
    2系統を持つとインターフェースが複雑化する（開発方針「シンプルに一つずつ」に反する）。
  - 影響: CLAUDE.md のコマンド例・phase4 のドキュメントを更新する必要がある。

## 1.6 優先順位（MoSCoW）

- **Must**: FR-001, FR-002, FR-003, FR-004, FR-005, FR-006
- **Could**: なし
- **Won't（本案件対象外）**: 複数カメラ一括、描画スタイルのカスタマイズ、dry-run
- **MVP/完成条件**: FR-001〜FR-006 すべて。MP4（FR-006）も本案件の完成条件に含める
  （ユーザー要求が「連番PNG + MP4」のため Must とする）。
  なお実装順序としては FR-001〜005（連番PNG）を先に動作確認し、続けて FR-006（MP4）を実装してよい。
