# 要求仕様書: feat-016 キーポイントのオクルージョン（深度による前後判定）

## 1. プロジェクト概要

### 1.1 何を作るのか
feat-015（`phase4/render_keypoints.py` のピンホール3DGSレンダリング）を拡張し、人体3Dキーポイント（C3D, Halpe26）の**先頭フレーム1枚**を、同じカメラ視点の3DGSレンダリング画像（PNG1枚）に重ね描きする。その際、**3DGSの深度マップと各キーポイントのカメラ深度を比較して前後関係（オクルージョン）を反映**し、カメラ手前の3DGS（障害物）に隠れるキーポイント・ボーンを隠蔽する。

### 1.2 なぜ作るのか
キャリブレーション（R, t, K）の妥当性を、人体キーポイントが3DGS点群上の正しい位置・姿勢に乗るかで目視検証したい。単純な2D重ね描きでは、カメラ手前の障害物に隠れるべきキーポイントが貫通して描かれてしまい、前後関係の正しさを判断できない。深度比較による隠蔽で、検証画像の妥当性を高める。

### 1.3 誰が使うのか
本プロジェクトの開発者（カメラパラメータ推定・3DGS検証を行う者）。

### 1.4 どこで使うのか
phase4 の uv 環境（CUDA GPU 必須、`TORCH_CUDA_ARCH_LIST="9.0+PTX"`）。`phase4/` ディレクトリで CLI として実行する。

## 2. 用語定義

機能設計書・コード内でも同じ用語を使う。

| 用語 | 定義 |
|---|---|
| キーポイント | 人体関節点。本機能では Halpe26（26点）を扱う |
| Halpe26 | 26点の人体キーポイント定義。C3Dのマーカー名と一致する |
| ボーン | 2つのキーポイントを結ぶ線分（スケルトンの辺） |
| C3D | モーションキャプチャの3D座標時系列フォーマット。本機能では py-c3d で読む |
| 先頭フレーム | C3Dの最初の有効フレーム（`first_frame` 起点の0番目）。本機能はこの1フレームのみ扱う |
| キャリブ座標 | PLY・カメラ外部パラメータと同一のワールド座標系（Z-up・メートル） |
| カメラ深度 | キーポイントのカメラ座標系Z成分 `(R·X+t)_z`[m]。光軸方向の距離 |
| 深度マップ | gsplat `render_mode="RGB+ED"` が返す画素ごとの期待深度（α加重期待値）[m] |
| αマップ | gsplat が返す画素ごとの累積不透明度（accumulation）。0〜1 |
| オクルージョン | カメラ手前の3DGSにキーポイントが隠れる前後関係。深度比較で判定する |

## 3. 機能要求一覧

### FR-001 C3D先頭フレームのHalpe26読み込み
- **概要**: C3Dファイルから先頭フレームのHalpe26（26点）の3D座標と有効フラグを取得する。
- **入力**: C3Dファイルパス（位置引数 `c3d_path`）
- **出力**: 26点のキャリブ座標 `(26,3)` と有効フラグ `(26,)`（メモリ上。ファイル出力なし）
- **受け入れ基準**:
  - C3Dの先頭フレームのみを読む（2フレーム目以降は読まない）
  - C3DのマーカーラベルからHalpe26の26名を引き、Halpe26順に並べる
  - 各点の有効フラグは「residual >= 0 かつ 座標にNaNを含まない」で判定する
  - C3DにHalpe26名が不足する場合は不足名を表示して終了コード1

### FR-002 キーポイントのピンホール投影とカメラ深度算出
- **概要**: 先頭フレームの26点を、指定カメラのK・R・tで歪みなし（ピンホール）投影し、各点の2D画素座標とカメラ深度を求める。
- **入力**: 26点のキャリブ座標 `(26,3)`、カメラ辞書（K, rvec, tvec）
- **出力**: 2D座標 `(26,2)`、カメラ深度 `(26,)`
- **受け入れ基準**:
  - 投影は `cv2.projectPoints(..., distCoeffs=None)`（歪みなし。背景3DGSがピンホールのため整合させる）
  - カメラ深度は `(R·X+t)` のZ成分
  - NaN座標があってもクラッシュしない（描画側で有効フラグによりスキップ）

### FR-003 3DGSの深度マップ・αマップ取得
- **概要**: 指定カメラ視点で3DGS（PLY）をピンホールレンダリングし、背景BGR画像に加えて深度マップとαマップを取得する。
- **入力**: gaussians（PLY読み込み済み）、カメラ辞書、`--near-plane`
- **出力**: 背景BGR `(H,W,3) uint8`、深度マップ `(H,W) float32`、αマップ `(H,W) float32`
- **受け入れ基準**:
  - feat-015 と同一のピンホール古典経路（`camera_model="pinhole"`, `with_ut=False`, `packed=True`, 歪みなし）
  - `render_mode="RGB+ED"` で深度を取得する
  - 深度マップはカメラ光軸方向のz深度で、キーポイントのカメラ深度と同じ単位で比較できる
  - 深度・α取得は `render_image` の既定OFFの追加引数（`return_depth`）で行い、関数レベルの後方互換を保つ（非機能要求「後方互換（限定）」参照）

### FR-004 深度比較によるオクルージョン判定
- **概要**: 各キーポイントについて、カメラ深度と深度マップ・αマップを比較し、可視（描画する）か隠蔽（描画しない）かを判定する。
- **入力**: 2D座標 `(26,2)`、カメラ深度 `(26,)`、有効フラグ `(26,)`、深度マップ、αマップ、深度マージン、`near_plane`
- **出力**: 可視フラグ `(26,)`
- **受け入れ基準**:
  - 有効フラグが False の点は不可視
  - **カメラ深度が非finite（NaN/inf）、または `near_plane` 以下（カメラ背面・至近）の点は不可視**（カメラ後方の点が投影されて誤描画されるのを防ぐ）
  - 投影位置が画像外（`0<=u<W` かつ `0<=v<H` を満たさない）の点は可視（隠す3DGSが画面内に無い）
  - 該当画素のαが閾値（内部定数 0.5）未満なら可視（3DGSが無い/薄い→隠すものが無い）
  - キーポイントのカメラ深度が「深度マップ値 + マージン」より大きい（奥）なら不可視
  - 判定順序: valid → 深度の有効性（finite かつ > near_plane）→ 画像外 → α → 深度比較。α判定を深度比較より**必ず先**に行う（低α画素の深度は不安定で信頼できないため）
  - ボーンの線分サンプル点にも同じ深度有効性判定を適用する

### FR-005 オクルージョン考慮の重ね描き
- **概要**: 背景画像のコピーに、可視なキーポイント（点）と可視なボーンを色分けして描画する。
- **入力**: 背景BGR、2D座標、有効フラグ、可視フラグ、（ボーン部分隠蔽用に）キャリブ座標・カメラ・深度マップ・αマップ・マージン
- **出力**: 重ね描き済みBGR画像 `(H,W,3) uint8`
- **受け入れ基準**:
  - 点は「有効 かつ 可視」のみ円で描画する（手前の3DGSに隠れる点は描かない）
  - ボーンは両端が有効な辺について、両端の3D座標を線分サンプリングし、各サンプルの可視判定を適用して**可視区間のみ**線を描く（障害物を横切るボーンが途中で切れる）
  - 色分け: 被験者の右半身=赤、左半身=青、体幹・顔=緑、キーポイント円=黄（BGR）
  - 左右は被験者本人の解剖学的左右（マーカー名 R*/L*）に従う（カメラ視点ではない）

### FR-006 PNG出力
- **概要**: 重ね描き済み画像をPNG1枚として保存する。
- **入力**: `--output`（出力パス）
- **出力**: PNGファイル1枚
- **受け入れ基準**:
  - `--output` 省略時は `./data/keypoints_<カメラ名>.png`
  - 出力ディレクトリが無ければ作成する
  - 保存失敗（`cv2.imwrite` の False 返却・`cv2.error` 送出）時はエラー表示・終了コード1

### FR-007 オクルージョン無効化オプション
- **概要**: 深度判定をOFFにし、全キーポイント・全ボーンを手前扱いで描画する（旧2D重ね描き相当。比較・デバッグ用）。
- **入力**: `--no-occlusion`（フラグ）
- **出力**: 全有効点・全有効辺を描画したPNG
- **受け入れ基準**:
  - `--no-occlusion` 指定時は `render_image` を `return_depth=False` で呼び、**深度マップ・αマップを生成しない**（深度レンダリング経路を通さない）
  - 可視判定をスキップし、点は有効のみ、ボーンは両端有効の辺を全描画する

### FR-008 深度マージン調整オプション
- **概要**: オクルージョン判定の深度マージン[m]をCLIで調整できる。
- **入力**: `--occlusion-margin`（float, 既定 0.05）
- **出力**: 判定に用いるマージン値の変更
- **受け入れ基準**:
  - 既定値 0.05m。3DGS深度ノイズ＋マーカーが体表より内側に埋まる分を吸収する

## 4. 非機能要求

- **対応環境**: phase4 uv 環境（Python 3.10）、CUDA GPU 必須、`TORCH_CUDA_ARCH_LIST="9.0+PTX"`。純粋ロジック（座標変換・投影・可視判定・描画）はルートvenvの pytest でCUDA非依存にテストできること
- **パフォーマンス**: 3DGSレンダリングは1回のみ（1フレーム1枚）。処理時間の明示的上限は設けない
- **信頼性**: 欠損キーポイント（residual<0/NaN）・隠れるキーポイントでクラッシュしない（描画スキップ）
- **後方互換（限定）**: 本案件は `render_keypoints.py` を「3DGSレンダリングのみ」から「キーポイント重ね描き専用CLI」に発展させる。`c3d_path` を必須位置引数として追加するため、**CLI互換は意図的に破る**（feat-015 の旧2引数CLIは廃止）。守る互換は次の2点に限定する:
  - `render_image` 関数の互換: `return_depth` 既定 False のとき feat-015 と完全に同一挙動（関数レベルの後方互換）
  - feat-015 の純粋ロジックテスト（`camera_to_viewmat`/`load_cameras_toml`/`select_camera`）は無変更で通る。`main` を呼ぶテスト3件は新CLIに合わせて更新する（壊さないのではなく更新対象）

## 5. 制約条件

- **使用必須**: gsplat（3DGSレンダリング）、OpenCV（投影・描画・PNG保存）、tomli（TOML）、c3d（C3D読み込み、phase4環境のみ新規追加）
- **使用禁止**: gsplat の UT経路（`with_ut=True`）＋歪み係数（feat-013/015 で黒い靄・品質劣化のため不採用。ピンホール統一）
- **依存追加**: `c3d`（PyPI py-c3d）を `phase4/pyproject.toml` と `docs/TECH_STACK.md` に追加する
- **データ**: PLY・TOML・C3D は gitignore 対象。実データ依存テストは存在時のみ実行（skipif）
