# feat-020 不具合調査

## イテレーション1: Blender上でフィルタ後C3Dが90度回転して見える（2026-07-02）

### 事象（ユーザー報告）

`filter_c3d.py` の出力C3D（`session001_world_22pt_filtered.c3d`）と入力C3D
（`session001_world_22pt.c3d`）を Blender（io_anim_c3d）で取り込み比較すると、
フィルタ後の3Dキーポイントがフィルタ前に対して約90度（符号不明）回転して見える。

### 調査内容と結果

調査対象: `phase4/filter_c3d.py`。以下の3段階で事実確認した。

1. **素通し検証（フィルタ無効化）**: `load_c3d` → `write_c3d` をフィルタを介さず
   直結し、入力C3D（68,389,376バイト）と出力をバイト比較 → **完全一致**。
   `filter_c3d.py` の読み書き経路（ヘッダ・パラメータ・座標エンコード）は無損失。
2. **実ファイルのメタデータ比較**: ユーザーが比較した実物の
   `session001_world_22pt_filtered.c3d` と入力の、ヘッダ＋パラメータセクション
   （データ開始ブロック7の手前、3,072バイト）をバイト比較 → **完全一致**。
   POINT:X_SCREEN(+Z) / Y_SCREEN(+Y) / UNITS(mm) / SCALE(-1.0) / rate(30.0) /
   first_frame(1) / ラベルすべて同一。相違は点データ部のみ（フィルタ適用なので正常）。
3. **点データの回転有無の実測**: 両ファイルの全有効サンプル（4,266,658点）で
   入力→フィルタ済の最適回転行列を Procrustes（SVD）で推定 → **単位行列、回転角
   0.000度**。座標差は平均6.1mm・中央値3.2mm（平滑化変位として妥当。最大709.7mm は
   スパイクノイズが均された箇所）。

### 結論

**`filter_c3d.py` に不具合はない。** 出力C3Dはメタデータが入力と完全一致し、
点データにも回転は存在しない。ファイルとしてはフィルタ前後で「座標が数mm滑らかに
なっただけ」の関係にある。

したがって、Blender上で見えた90度回転はC3Dファイルの内容由来ではなく、
**Blenderへの取り込み手順・io_anim_c3d 側の要因**と考えられる。候補:

- 取り込み時のインポートオプション（軸設定・回転）の差
- 片方のアーマチュア/Emptyにオブジェクトレベルの回転が掛かっている
  （既存シーンに取り込んだ場合、既存オブジェクトの変換との差）
- io_anim_c3d の取り込み向きは pose bone の rest 行列にも依存する（feat-018 で確認済み）

### 解決（2026-07-02 ユーザー確認）

Blenderの新規ファイルにフィルタ前・フィルタ後の2つのC3Dを同一手順でインポートし、
最初のフレームで各キーポイントの位置が一致することをユーザーが確認した。
**フィルタ済みC3Dは回転していない。** 当初の90度回転は最初の取り込み手順に起因する
見かけ上のものであり、`filter_c3d.py` の不具合ではない。コード修正なしでクローズ。
