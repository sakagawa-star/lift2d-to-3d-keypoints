# 要求仕様書: feat-013 3DGSレンダリング画像への3Dキーポイント重ね描き

## 1. 背景・目的

Blender 可視化用にインポートしている3つの元データ（LCC Studio製の3DGS PLY、Pose2Sim製のキャリブレーションTOML、Pose2Sim製の人体キーポイントC3D）を使い、指定した1台のカメラ視点で3DGSをレンダリングし、その画像上に人体3Dキーポイント（Halpe26）を投影し、**3DGSとの3次元的な前後関係（オクルージョン）を反映して**点＋ボーンで重ね描きする。連番PNGまたはMP4で出力する。手前の3DGS（障害物）に隠れるキーポイントは隠蔽されることが、本機能の中核要件である。

3D Gaussian Splatting の見た目（実空間の再構成）の上に Pose2Sim の推定キーポイントを重ねることで、キーポイント推定結果やキャリブレーション結果を3D空間整合の取れた形で目視確認・記録できるようにする。約1時間分（30fps）の長尺データの静止画化・動画化を想定する。

**重要な設計方針**: Blender は確認用であり、本機能は Blender を一切経由しない。元データ（PLY / TOML / C3D）から直接、OpenCV / gsplat の座標系で処理する。

## 2. 入力

実データは `phase4/data/Blender/` に存在する。

### 2.1 PLY（3DGS点群）
- LCC Studio（XGRIDS PortalCam）製。INRIA標準3DGSフォーマット
- 座標系: Z-up・右手系・メートル単位。**ネイティブ座標がそのままキャリブレーション座標系**（キャリブの外部パラメータがLCC点群由来の基準点で求められているため）
- `render.py` の `load_ply()` で読み込める形式（`x,y,z`, `scale_*`, `rot_*`, `opacity`, `f_dc_*`, `f_rest_*`）

### 2.2 TOML（カメラキャリブレーション）
- Pose2Sim Calib 形式。複数カメラのテーブルを含む（例: `[cam41520554]`）
- 各カメラ: `name`, `size`=[W,H], `matrix`=K(3x3), `distortions`=[k1,k2,p1,p2,k3], `rotation`=Rodriguesベクトル(3), `translation`=t(3), `fisheye`
- R（Rodrigues→行列）, t は **OpenCV規約の world-to-camera**（カメラ座標 = R·X_world + t、+Z前方・Y下向き）
- カメラ判定は `matrix` キーの有無で行う（防御的）。実データ `Config_scene.toml` には `metadata` 等の非カメラテーブルは無く5テーブル全てがカメラだが、Pose2Sim TOML一般では `metadata` 等が混じりうるため除外ロジックを持つ

### 2.3 C3D（人体キーポイント）
- Pose2Sim 製。フレームごとの3Dマーカー座標
- 単位: mm（`POINT:UNITS = mm`）
- 含まれるマーカー: Halpe26（26点）＋ LSTM拡張 `_study` マーカー（43点）の計69点。本機能は **Halpe26 の26点のみ** を使う
- マーカーは名前（label）で識別する（順序に依存しない）
- フレームレート: 30fps（実データ）だが、カメラ設定で変わりうる。C3Dの `point_rate` から取得する

## 3. 座標系の確定事項（調査・検証済み）

本機能の成否を分ける座標変換は、実データでの再投影検証により以下に確定している。

### 3.1 C3D生座標 → キャリブレーション座標
C3Dの生座標 `p = (px, py, pz)`（mm）に対し:

```
X_calib = (pz, px, py) × 0.001   （単位 mm → m）
```

- これは Pose2Sim の `zup_to_yup`（triangulation後のZ-up→Y-up変換: 新座標=(Y,Z,X)）の逆変換に一致する
- 検証: 5台すべてのカメラに対し、フレーム1のHalpe26 26点を `cv2.projectPoints` で再投影したところ、各カメラで24〜26点が画像枠（1920×1080）内、全点がカメラ前方（Z>0）、頭（Nose）が足（RAnkle）より上、という妥当な結果を得た。誤った候補変換 `(pz,py,-px)` 等では枠外・逆さ・座標発散となり明確に棄却された

### 3.2 カメラの座標変換
- gsplat 用 view行列（world-to-camera, OpenCV規約）は TOML から **直接** 構成する: `viewmat = [[R, t], [0,0,0,1]]`（R=Rodrigues(rotation), t=translation）。Blender経由の `c2w` 変換（`render.py` の `blender_to_opencv_c2w` / `rotate_z90`）は使わない
- PLY のネイティブ座標とこの viewmat は同じワールド座標系（キャリブ座標）で整合する

## 4. 機能要件

### 4.1 カメラ選択
- `--camera <名前>`（必須）で TOML 内の対象カメラを1台指定する
- 指定名が TOML に存在しない場合、エラーメッセージと利用可能なカメラ名一覧を表示し、終了コード1で終了する

### 4.2 キーポイント投影
- C3Dの各フレームのHalpe26マーカーを 3.1 の変換でキャリブ座標に変換し、`cv2.projectPoints(X, rvec=rotation, tvec=translation, K=matrix, distCoeffs=None)` で2D画素座標を得る（**歪みなし＝ピンホール**。背景の3DGSレンダリングがピンホールのため整合させる）
- あわせて各キーポイントの**カメラからの深度**（`(R·X + t)` のZ成分）を求める。オクルージョン判定に用いる
- マーカーの有効判定は **C3Dの residual 列（`points[:, 3]`）に基づく**。`residual < 0` は無効サンプル（カメラ未観測）であり、座標にNaNを含む点とあわせて投影・描画しない。「全成分0」は判定基準にしない（原点近傍の正当な点を誤って無効化するため）

### 4.3 重ね描き（オクルージョン対応）
- **点**: 各Halpe26キーポイントを円で描画する
- **ボーン**: Halpe26スケルトン定義（design.md 参照）に従い、両端の点がともに有効な辺のみ線分で描画する
- 左半身・右半身・体幹/顔で色を分けて視認性を確保する
- **オクルージョン（3次元的隠蔽、本機能の中核）**: 各キーポイントのカメラ深度と、投影先画素の3DGS深度（深度マップ）を比較し、**キーポイントが手前の3DGSより奥にある（隠れる）場合は描画しない**。これにより、カメラ手前の障害物に人体が一時的に隠れる動きを正しく表現する。ボーンの部分的な隠蔽の扱いは design.md で定義する
- 画像枠外に投影された点・線も描画してよい（OpenCVのクリッピングに委ねる）

### 4.4 レンダリング
- gsplat で 3DGS を 4.1 のカメラ・TOMLの `size` の解像度でレンダリングする
- **歪みなし（ピンホールカメラモデル）**: 標準3DGSはピンホール専用で、gsplatのUT歪み（`with_ut=True`）は投影の近似のため品質が劣化（黒い靄・アーティファクト）する。`camera_model="pinhole"`・歪み係数なし・`with_ut=False`・`packed=True`（= `render.py` と同じ古典経路）でレンダリングする。背景もキーポイント投影も歪みなしで整合する。実カメラのレンズ歪みは反映しない（歪み補正済み画像相当の見え方）
- **深度マップを取得**: オクルージョン判定のため `render_mode="RGB+ED"` で各画素の深度（カメラからの距離）も出力する
- 主点（cx,cy）が画像中心からずれていても TOML の K をそのまま使い正しく反映する

### 4.5 出力
- デフォルト: 連番PNG（`frame_<絶対フレーム番号>.png`、6桁ゼロ詰め）を出力ディレクトリに `cv2.imwrite`（BGR）で保存する
- `--mp4`: MP4動画として出力する（`render.py` 同様、NVENC利用可能ならNVENC、なければlibx264。入力pix_fmtはBGR描画に合わせ `bgr24`）
- `--mp4-fps`: MP4フレームレート（未指定時はC3Dの `point_rate` を四捨五入）
- `--dry-run`: 画像を保存せず、レンダリング・投影・描画の動作確認と速度計測のみ行う
- `--start-frame` / `--end-frame`: 処理するC3Dフレーム範囲を **C3D絶対フレーム番号**（`first_frame` 起点、両端含む）で指定する（長尺データの部分処理用）
- `--background`: 背景色RGB[0-1]（`render.py` 踏襲、デフォルト黒）
- `--no-keypoints`: キーポイント描画なし（背景のみ。デバッグ用）。カメラ固定のため全フレーム同一背景になる旨を警告表示する
- `--no-occlusion`: オクルージョン判定を無効化し、全キーポイントを常に手前に描く（旧2D重ね描き相当）。デバッグ・比較用
- `--occlusion-margin`: オクルージョン判定の深度マージン[m]（既定 0.05。キーポイントが3DGSよりこの値以上奥なら隠す）
- 出力ディレクトリは `--output-dir`（デフォルトは `render.py` と同様 `./data/images`）

### 4.6 フレーム対応
- カメラは固定（全フレーム同一ポーズ）。C3Dのフレームごとにキーポイントが変わる
- 3DGS点群は静的・カメラ固定のため、**背景レンダリングは1回のみ**実行して全フレームで使い回す（長尺データの高速化）
- 出力フレーム番号はC3Dの絶対フレーム番号（`first_frame` 起点）に対応させる。PNG名は `frame_{絶対番号:06d}.png`（`render.py` 踏襲）

## 5. 出力仕様

- 連番PNG または MP4。画像サイズは TOML の `size`（実データは1920×1080）
- 各フレーム: gsplatレンダリング画像（ピンホール）の上に、3DGSとの前後関係（オクルージョン）を考慮してHalpe26の点＋ボーンを重ねたもの
- 実行時に対象カメラ名・総フレーム数・処理フレーム数・所要時間（秒/フレーム）を表示する

## 6. 品質基準

- 指定カメラのキャリブで、3DGSレンダリングとキーポイント投影が目視で整合する（人体が点群上の妥当な位置・姿勢に重なる）こと。**カメラ手前の3DGSに隠れるキーポイントが正しく隠蔽される**こと。実機でのレンダリング結果で確認する
- 存在しないカメラ名でエラー終了すること
- 欠損キーポイントを含むフレームでもクラッシュせず、有効点のみ描画されること
- 純粋ロジック（C3D→キャリブ座標変換、TOMLカメラのview行列構成、Halpe26マーカー名フィルタ、欠損判定、キーポイントのカメラ深度計算、オクルージョン判定）について bpy / CUDA 非依存のユニットテストを `tests/` に追加し、`uv run pytest -v` が全て成功すること
- gsplat レンダリング・実画像出力部分は実機（GPU + `TORCH_CUDA_ARCH_LIST="9.0+PTX"`）での手動テストで確認する

## 7. 対象外（スコープ外）

- 複数カメラの同時出力（1回の実行で1カメラ）
- 動くカメラ（カメラアニメーション）への対応（固定カメラ前提）
- _study マーカー43点の描画（Halpe26のみ）
- C3Dフレームレートとカメラ/出力フレームレートの厳密な同期・リサンプリング（C3Dフレーム＝出力フレームの1対1対応のみ）
- キーポイントの信頼度・可視性スコアによる表示制御（C3Dに含まれないため）
- レンズ歪みの再現（ピンホールで統一。実カメラのレンズ歪みは反映せず、歪み補正済み画像相当の見え方になる。3DGSにUT歪みを乗せると靄・品質劣化が出るため不採用）
- Blender経由のワークフロー（`camera_pose.py` → poses JSON）との統合
- 3DGSレンダリングと実カメラ映像の厳密な一致（3DGSの再構成品質・SH次数・露出差は対象外）
- 半透明オクルージョン（3DGSが半透明な箇所でキーポイントを部分的に透かす等）。判定は深度のしきい値による二値（隠す/隠さない）とする
