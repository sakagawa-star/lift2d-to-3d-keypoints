# feat-021 要求仕様書: render_keypoints.py 欠損マーカー許容（22点C3D対応）

## 1.1 プロジェクト概要

- **何を作るのか**: `phase4/render_keypoints.py` を、C3D に Halpe26 の全26マーカーが
  揃っていなくても実行できるように変更する。存在するマーカーのみ描画し、
  `Spine`/`Thorax` マーカーを通る体幹ボーンにも対応する。
- **なぜ作るのか**: リフトアップ推定由来の22点C3D（足先6点なし、`Spine`/`Thorax` あり）を
  レンダリングしたいが、現状は「Halpe26マーカー不足」の `ValueError` で実行できない。
- **誰が使うのか**: 本プロジェクトの開発者（キャリブレーション・リフトアップ結果の目視検証者）。
- **どこで使うのか**: gtune2（Linux、RTX 5060 Ti、CUDA 12.6）の phase4 uv 環境。
  `TORCH_CUDA_ARCH_LIST="9.0+PTX"` を付けて CLI 実行する。

## 1.2 用語定義

機能設計書・コード内でも同じ用語を使う。

| 用語 | 定義 |
|---|---|
| Halpe26 | 26点の人体キーポイント定義。マーカー名は render_keypoints.py の `HALPE26_NAMES` の26名 |
| 22点C3D | Halpe26 から足先6点（RBigToe, RSmallToe, RHeel, LBigToe, LSmallToe, LHeel）を除き、Spine, Thorax を加えた22マーカーのC3D。`phase4/npz_to_c3d.py` の出力形式 |
| 既知マーカー | 本スクリプトが名前を認識して描画対象にするマーカー。Halpe26 の26名 + Spine + Thorax の計28名 |
| 欠損マーカー | 既知マーカーのうち、入力C3Dのラベルに存在しないもの |
| 無効サンプル | C3D 上でマーカーは存在するが、当該フレームで residual < 0 または座標に NaN を含むサンプル |
| ボーン | 2つのマーカーを結ぶ線分（スケルトンの辺） |
| 体幹フォールバックボーン | Neck–Hip の直結ボーン。Spine と Thorax の両方が欠損マーカーの場合のみ描画する |

## 1.3 機能要求一覧

### FR-001: 欠損マーカーを許容した実行

- **機能名**: 欠損マーカー許容
- **概要**: 入力C3Dに既知マーカーの一部が存在しなくても、エラー終了せずレンダリングを実行できる。
- **入力**: 既知マーカーのうち1個以上を含む C3D ファイル（位置引数 `c3d_path`）。
- **出力**: 存在する既知マーカーのみを点として描画した連番PNG（および `--mp4` 指定時はMP4）。
  欠損マーカーは全フレームで点・ボーンとも描画されない。
- **受け入れ基準**:
  1. 22点C3D（`session001_world_22pt_filtered.c3d` と同構成）を入力すると、エラーなく
     終了コード 0 で完走し、PNGが出力される。
  2. 26点C3D（Halpe26完備、Spine/Thorax なし）入力時の描画内容が変更前と同一であること。
     検証は pytest（合成データ）で行う:
     (a) `build_skeleton`（26点 present）の結果が変更前の `HALPE26_SKELETON`
     （25本・同順）と完全一致すること。
     (b) `extract_keypoints` の先頭26点の座標・valid が変更前 `extract_halpe26`
     の結果と同値であること。
     加えて、実データの26点C3Dが利用可能な場合は変更前後の出力PNGのバイト一致を
     手動テストで確認する（実データが無い場合はこの手動確認は省略可。pytest の
     (a)(b) が必須の受け入れ基準）。

### FR-002: 端点が欠けるボーンのスキップ

- **機能名**: ボーンの欠損スキップ
- **概要**: ボーンの両端点のうち一方でも欠損マーカーの場合、そのボーンは全フレームで描画しない。
  （無効サンプルによるフレーム単位のスキップは既存挙動のまま。）
- **入力**: FR-001 と同じ C3D。
- **出力**: 両端点とも C3D に存在するボーンのみが描画された画像。
- **受け入れ基準**: 22点C3Dの出力画像に、足先6点に接続するボーン6本
  （RAnkle–RHeel, RAnkle–RBigToe, RBigToe–RSmallToe, LAnkle–LHeel, LAnkle–LBigToe,
  LBigToe–LSmallToe）が描画されない。

### FR-003: Spine/Thorax を通る体幹ボーン

- **機能名**: 体幹ボーン拡張
- **概要**: C3D に `Spine` と `Thorax` が存在する場合、体幹を
  Neck–Thorax–Spine–Hip の3本のボーンで描画する。両方とも欠損の場合は
  従来の Neck–Hip 直結（体幹フォールバックボーン）を描画する。
  片方のみ存在する場合は、存在する側を経由する2本で描画する
  （Thorax のみ: Neck–Thorax, Thorax–Hip。Spine のみ: Neck–Spine, Spine–Hip）。
- **入力**: FR-001 と同じ C3D。
- **出力**: 上記規則に従った体幹ボーン（色は既存の体幹色 = 緑）と、
  Spine/Thorax の点（既存の点色 = 黄）。
- **受け入れ基準**:
  1. 22点C3Dの出力画像で体幹が Neck–Thorax–Spine–Hip の3本で描かれ、
     Neck–Hip 直結ボーンが描画されない。
  2. 26点C3D（Spine/Thorax なし）では Neck–Hip 直結ボーンが描画される（FR-001 基準2に包含）。

### FR-004: マーカー構成の起動時報告

- **機能名**: マーカー構成レポート
- **概要**: 起動時（C3D読み込み直後）に、既知マーカーの充足状況を標準出力へ表示する。
- **入力**: FR-001 と同じ C3D。
- **出力**: 標準出力に以下の2行（欠損が0個の場合は2行目を出力しない）:
  - `キーポイント: <存在数>/<既知マーカー総数> マーカーを描画対象とします`
  - `欠損マーカー（描画スキップ）: <カンマ区切りの欠損マーカー名>`
- **受け入れ基準**: 22点C3Dで「22/28」と足先6点の欠損名が表示される。
  26点C3Dで「26/28」と Spine, Thorax の欠損名が表示される。

### FR-005: 既知マーカーが不足する場合のエラー

- **機能名**: 最低マーカー数エラー
- **概要**: 入力C3Dのラベルに既知マーカーが1個も存在しない場合、描画する対象がないため
  エラーメッセージを表示して終了コード 1 で終了する。
- **入力**: 既知マーカーを1個も含まない C3D。
- **出力**: 標準エラー出力にエラーメッセージ
  `C3Dに既知マーカーが1つもありません（labels: <C3Dのラベル一覧>）`。終了コード 1。
- **受け入れ基準**: 既知マーカー0個のC3Dを与えると、PLY読み込み（torch/gsplatロード）より
  前に上記メッセージで終了する。

## 1.4 非機能要求

- **パフォーマンス**: 26点C3D入力時のフレームあたり処理時間は変更前の110%以内
  （マーカー数・ボーン数が変わらないため実質同等）。22点C3D入力時は描画対象が減るため
  変更前26点時と同等以下。
- **対応環境**: 既存と同一（Linux、phase4 uv 環境、CUDA GPU 必須、
  `TORCH_CUDA_ARCH_LIST="9.0+PTX"`）。
- **信頼性**: 既存のエラーハンドリング（PNG保存失敗・ffmpeg失敗時の終了コード 1、
  ffmpeg クリーンアップ保証）を変更しない。
- **セキュリティ**: 該当なし。

## 1.5 制約条件

- 使用ライブラリは既存のまま（numpy, opencv-python, tomli, c3d, torch, gsplat）。
  新規ライブラリの追加は行わない。
- CLI インターフェース（位置引数・オプション）は変更しない。オプションの追加も行わない。
- `phase4/render.py`, `phase4/npz_to_c3d.py`, `phase4/filter_c3d.py` は変更しない。
- 変更対象ファイルは以下の3ファイルに限定する:
  - `phase4/render_keypoints.py`（本体）
  - `tests/test_feat016_keypoints.py`（`extract_halpe26` リネーム・`draw_overlay`
    シグネチャ変更・欠損時挙動変更（ValueError → valid=False）への追随。
    旧APIのまま残すと pytest が import error / TypeError で壊れるため）
  - `tests/test_feat021_flexible_markers.py`（新規テスト）
