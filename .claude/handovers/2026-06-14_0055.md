## 決定事項

- **CLI**: `--camera`（必須）/`--near-plane`（既定0.1）/`--output`（既定 `./data/render_<カメラ名>.png`）/`--background`（既定 黒）
- **viewmat**: TOML の rotation(Rodrigues)/translation から `[[R,t],[0,0,0,1]]` を直接構成（OpenCV world-to-camera。Blender変換は挟まない）
- **gsplat経路**: `camera_model="pinhole"`・`with_ut=False`・`packed=True`（render.py の render_frame と同一の古典経路）。歪み・UT・深度は使わない
- **PNG保存失敗は2系統を扱う**: `cv2.imwrite` の `False`返却（権限・エンコード失敗）と `cv2.error`送出（不正な拡張子・拡張子なし）。両方とも exit1。OpenCV 4.13.0 で実挙動確認済み
- **main の処理順**: `select_camera` を `from render import` より前に置く（誤カメラ名の早期検出＋ルートvenvで不在カメラ経路を非skipテスト可能にするため）
- **GTより左右視野がわずかに狭い件は仕様内として完了**（ユーザー判断）。ピンホールで歪みを再現しない（要求スコープ外）こと等に起因と推定
- **Codexレビューはサンドボックス回避フラグ不要**（環境側で解消済み。通常の `codex exec "..."` で動作）

## 捨てた選択肢と理由

- **gsplatのUT経路で歪みを再現**: 黒い靄・品質劣化が出るため不採用（feat-013で確認済み、ピンホール統一）
- **`cv2.imwrite` の戻り値チェックのみ**: OpenCV 4.13.0 では不正拡張子で `cv2.error` を送出するため、戻り値チェックだけだと traceback クラッシュ。try/except と両用に
- **Codex `--dangerously-bypass-approvals-and-sandbox`**: 一時はbwrap loopbackエラー回避に使ったが、環境側で不要になったため使わない（CLAUDE.mdへの追記も見送り）
- **コミットを feat-015 だけに分離**: CLAUDE.md が前セッションの全面改訂と混在しファイル単位で分離不可。ユーザー判断で全変更を1コミットにまとめた

## ハマりどころ

- **Codexのbubblewrapサンドボックス**が `bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted` で全ファイル読取コマンド失敗 → 後に環境側で解消（Codex v0.135.0→v0.139.0 のバージョン差が関係した可能性）
- **`render.py` は torch を top-level import**しており、ルートvenv（pytest）では `import render` が失敗する → main テストでは `sys.modules["render"]` にダミーモジュール（load_ply/print_ply_summary スタブ）を仕込み、`render_keypoints.render_image` も monkeypatch でスタブ化して保存直前まで到達させる
- **旧 test_feat013 は untracked だった**ため、削除しても git status に出ない（コミット対象外）

## 学び

- `cv2.imwrite` は失敗時に `False`返却と例外送出の2系統がある（OpenCV 4.13.0）。エラーハンドリングは両方必須
- Codexレビューは実機でライブラリ挙動を検証できる（今回 `cv2.imwrite` の例外挙動を実証）。サンドボックス内で動く分、推測でなく事実ベースの指摘が得られる
