## 決定事項

- **座標規約**: world `(X,Y,Z)` m → C3D raw `(Y,Z,X)×1000` mm。`render_keypoints.c3d_to_calib`（raw(px,py,pz)mm→(pz,px,py)×0.001m）で復元可能（読み戻し誤差 6e-8m）。X/Y_SCREEN は render_keypoints の生mm直読みに影響しない（2消費者を独立に満たす）
- **関節**: NPZ 22ラベルをそのまま `POINT:LABELS` に書く（Halpe26整形しない）。`MANUFACTURER:SOFTWARE` 未設定（ラベル間引き回避）
- **C3Dフレーム番号**: 1始まり連番（1..F）固定。絶対 `frame_ids`（145599〜）は埋め込まずログ出力
- **Blender正立**: `UNITS='mm'` / `X_SCREEN='+Z'` / **`Y_SCREEN='+Y'`**
- **residual**: 座標が有限なら0.0（有効）、NaN/Inf なら-1.0（無効）。pnp_ok は未使用
- **安全策**: 出力拡張子`.c3d`必須・入力NPZと同一パス拒否・一時ファイル→読み戻し検証→`os.replace` のアトミック確定
- **CLI**: `npz_to_c3d.py <npz_path> [--output PATH] [--fps 30.0]`（`allow_abbrev=False`）

## 捨てた選択肢と理由

- **絶対フレーム番号をC3Dに保持（`set_start_frame(frame_ids[0])`）**: py-c3d 0.6.0 のヘッダ first/last frame が16bitで、65535超（145599）を渡すと読み戻し破綻（実測 first=145595/count=301/EOF）。→1始まりに変更
- **py-c3d を vendor/patch して32bitフレーム対応**: render_keypoints の読み取りも巻き込む大改修で「小さく作る」方針に反する。範囲外（必要なら別案件）
- **world をそのまま×1000（軸保存）でmm化**: render_keypoints 側に別途軸調整が必要で二重管理になる。→軸入替(Y,Z,X)を採用
- **Halpe26 26点へ整形**: Spine/Thorax破棄＋足6点欠損補完が必要で目的（Blender取り込み）に過剰
- **Y_SCREEN='-Y'（world鉛直=Y前提）/ '+X'**: いずれも実機で上下逆（下記ハマりどころ）

## ハマりどころ

- **軸の向きを2回外した（最重要）**:
  1. spec §4 本文「y=垂直」を鵜呑みにし `Y_SCREEN='-Y'` → 上下逆。実データの鉛直軸は実測で **Z**（Head−Ankle が Z軸で+0.78、全フレーム頭が+Z側）だった（注記「←キャリブ依存」が正しい）
  2. `Y_SCREEN='+X'` に修正 → **まだ上下逆**。原因は **`io_anim_c3d` が pose bone のローカル座標として描画する**点を見落としていたこと。+Z向きボーンの rest 行列 `B=Rx(+90)=[[1,0,0],[0,0,-1],[0,1,0]]` がもう一段座標を入れ替え、表示ワールド = `B @ global_orient @ p_raw`。**表示鉛直は `Y_SCREEN`(axis_y) だけで決まる**。`B`込みモデルが過去2回の実測を再現し `Y_SCREEN='+Y'` で正立と確定
- 教訓: `axis_interpretation` を手計算で再現して検証すると `B` を見落とす。**向き検証は実物アドオンコードを直接実行**すること（`__init__` の bpy import はパッケージスタブで回避可能。T-3テストがこの方式）
- 単位スケール: `_displayed_world` で `unit_conversion`（mm→m, ×0.001）を掛け忘れて T-3 が一度失敗（向き・符号は正しかった）
- 警告 `UserWarning: No analog data found in file.` はアナログ無しC3Dの正常挙動。**A（そのまま）**で対応決定（既存 render_keypoints 経路でも出る）

## 学び

- io_anim_c3d の取り込み向きは X/Y_SCREEN だけでなく **pose-bone rest 行列**も効く（メモリ `reference_io_anim_c3d_bone_rest_matrix` に保存）
- py-c3d 0.6.0 のフレーム番号は16bit制約あり。大きい絶対フレーム番号は1始まりに正規化が無難
- spec の座標軸記述は「キャリブ依存」注記がある場合、実データで鉛直軸を実測確認すべき
