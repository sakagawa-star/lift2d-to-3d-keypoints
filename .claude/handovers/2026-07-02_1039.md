## 決定事項

- フィルタは 2次Butterworth + `scipy.signal.filtfilt`（実効4次・ゼロ位相）固定。`--cutoff` 既定 6.0 Hz
- サンプリング周波数はC3Dの point rate から取得。`--rate` は rate 欠損時の補完専用（有効rateとの併存はエラー）
- 無効サンプル: `--max-gap`（既定10フレーム）以下のギャップのみ線形補間、超過はセグメント分割して独立フィルタ。出力では無効を無効のまま維持
- 入力は本プロジェクト規約のC3D（mm / X_SCREEN=+Z / Y_SCREEN=+Y、first_frame 1〜65534）に限定し、規約外（軸パラメータ欠落含む）はエラー終了
- 入出力同一判定は realpath + samefile（symlink経由の同一実体も拒否）
- phase4 に scipy>=1.11 追加（1.15.3 導入）
- テスト実行コマンド: `uv run --project phase4 --with pytest pytest tests/test_filter_c3d.py -v`

## 捨てた選択肢と理由

- **fps_camera_pose.py 内でボーン位置を平滑化** → カメラだけ滑らかになり、映像内の人体（アーマチュア・render_keypoints描画）のカクつきが残るため却下
- **npz_to_c3d.py に --smooth オプション追加** → 既存C3D・他所由来C3Dに適用できないため却下
- **回転を計算後に平滑化（案A）** → クォータニオン符号反転等の複雑さ。キーポイント（源）を平滑化する方が素直
- **Savitzky-Golay / 移動平均 / One Euro Filter** → パラメータ2個 / 動きが鈍る / リアルタイム向けでオフラインに利点なし
- **無制限ギャップ補間** → 長い欠損の直線補間値が filtfilt 経由で有効サンプルを汚染（Codex指摘で棄却）
- **スクリーン軸欠落の許容** → 規約外C3Dが素通りし出力で規約値が付与され姿勢が黙って変わる（Codex指摘で棄却）
- **first_frame の無制限引き継ぎ / 1固定** → py-c3d 16bit制約で65535以上は破綻 / 他所由来C3Dの対応を壊す → 1〜65534 のみ許可・範囲外エラーに

## ハマりどころ

- 手動テストで「フィルタ後C3Dが90度回転」報告。調査手順（ユーザー提案の素通し検証が決定打）:
  1. フィルタ無効の load_c3d→write_c3d 素通しで入力と**バイト完全一致** → 読み書き経路は無損失
  2. 実物フィルタ済みファイルのヘッダ+パラメータ部（3,072バイト）も入力と完全一致
  3. 全有効サンプル426万点のProcrustes回転推定 → 0.000度
  → 原因はBlender取り込み手順側。新規ファイルへの再取り込みで解消を確認

## 学び

- 「ファイルが壊れている疑い」には素通し（identity変換）でのバイト比較が最強の切り分け。フィルタや変換のロジックを疑う前に I/O 経路の無損失性を証明できる
- Codexレビューはメタデータの「維持」と「固定値書き込み」の意味論的矛盾のような、自分では見落とす整合性問題を拾う。今回は4サイクルで収束（高2→高1→中1→ゼロ）
- filtfilt の適用には時系列長 > padlen（2次で9）が必要。セグメント分割設計では最小長チェック（MIN_FILTER_SAMPLES=10）が必須
